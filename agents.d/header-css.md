Read before touching `css/header.css`, `js/header.js`, or `js/admin-bar-offset.js`.

Digest of `README.md` §Extending and §Why you can't simplify this, which hold the full rationale and the measurements. Change one, change both.

- Contract = 2 classes: `has-transparent-header` (user adds, on Template Part block), `is-scrolled` (plugin adds, past 20px).
- Transparent = **absence** of `is-scrolled`. Never an `is-transparent` class: a script-set class cannot exist at first paint → header flashes solid, then fades. Never invert.
- Transparent-mode rules are scoped to `header.has-transparent-header`. Three Ollie fixes are not, **on purpose** – the defects hit sticky headers regardless of transparency, so scoping them to the class re-breaks solid headers. Never ‘tidy’ them under the class. Two (`--sticky-full-offset`, the transition list) key off `.wp-block-group[data-sticky-on-scroll-up="true"]` and reach every sticky header, transparent or not. The third (the ≤600px `top: 0` reset) targets `header:not(.has-transparent-header):has(> [data-sticky-on-scroll-up])` – every *solid* sticky header, excluding the transparent one (which is `fixed` and sets its own `top`). The `<header>` tag requirement comes from the transparent-mode rules and Ollie's own sticky rule; the first two fixes don't name `header`, and the third names it only to reach Ollie's existing solid headers, not to impose anything new.
- Transparent-state rule keeps both its `!important` (core emits the block's own preset background as `.has-<slug>-background-color { background-color: … !important }` – nothing weaker beats it) and its `:not(:has([aria-expanded="true"]))` clause (an open menu drops the header to solid, so the panel gets a backdrop). Phrased ‘nothing is open’, never ‘something is closed’: the latter needs a closed element to exist.
- Never write the `transition` shorthand on the header group. Ollie Pro sets `transition: transform …` at (0,2,0); a shorthand landing there resets it → header snaps, not slides. `transition-property` longhand only.
- Stylesheet must load after the theme's: enqueue priority 20, dep on the `ollie` handle. `position: fixed` ties Ollie's sticky rule at (0,2,2) – source order alone decides.
- The `@media (max-width: 600px) { … top: 0 }` rule fixes a **theme** specificity bug (in Ollie's `style.css`), not an Ollie Pro one. Ollie pins every sticky header down by the admin bar's height and ships its own ≤600px reset to `top: 0` – but that reset (specificity 0,1,1) is outweighed by its own unscoped base rule `body:not(.wp-admin) header:has(> .is-position-sticky)` (0,2,2), and media queries add no specificity, so the reset is dead code: the header rests 46px low once the now-`absolute` admin bar has scrolled out of view at ≤600px. Our rule restores the reset with a selector heavy enough to win (0,4,2). It stays pure CSS because a `sticky` header makes `top: 0` correct at both ends: in the flow at the very top it still tucks under the bar, and it sits flush once stuck. It excludes `.has-transparent-header` on purpose – those are `fixed`, not `sticky`, and a static `top: 0` would misbehave for them (empty band at 46px, or bar-overlap at 0px), so the transparent header's ≤600px gap is fixed separately (next bullet). Verified on `/partners/` at 400px, logged in.
- The transparent (`fixed`) header has the **same** ≤600px gap, but can't use the static `top: 0` reset: out of the flow, it would overlap the bar at the very top instead of tucking under it. So `js/admin-bar-offset.js` measures the admin bar's live bottom edge (`getBoundingClientRect().bottom`, clamped ≥0) into `--kntnt-admin-bar-offset`, and the `fixed` rule reads `top: var(--kntnt-admin-bar-offset, var(--wp-admin--admin-bar--height, 0px))` – so it tucks under the bar at the top and sits flush once the bar scrolls away. Reading scroll position is the one thing CSS can't do, which is why JS feeds it. The script is its **own file**, enqueued only when `is_admin_bar_showing()` (logged-in) and attaching its scroll listener only below 600px (`matchMedia`), clearing the variable above 600px so the fallback owns `top`. Don't merge it into `header.js`: that file's `is-scrolled` job runs for every visitor, so merging would ship admin-only code to everyone and into the cache-minified public bundle. The fallback keeps the static height before the script runs, with no JS, and for logged-out visitors (who have no bar to clear). Verified on the home page at 400px against a faithful admin-bar mock. Anchor positioning (`top: max(0px, anchor(--wpadminbar bottom, 0px))`) is the future pure-CSS replacement – switch once stable iOS Safari ships it.
- Requires *Hide in Scroll Down* (`data-sticky-on-scroll-up="true"`) on the group. Without it Ollie Pro's JS writes an inline `position` on the `<header>`, which beats the plugin's non-`!important` `fixed` rule → header never goes transparent. This is why the setup contract makes that toggle mandatory, and part of why the `fixed` rule can't just add `!important`.
- Two rules handle the **Ollie Menu Designer mega menu** over a transparent header, both `@media (min-width: 600px)` (Ollie's own desktop threshold – the mobile menu is a separate full-screen overlay, not a panel hanging off the header) and both scoped to `header.has-transparent-header`. They are tightly coupled: fixing either alone changes what you see, so change them together.
  - **Fade sync.** Overrides the panel's opacity `transition` to `0.3s cubic-bezier(0.4, 0, 0.2, 1)` (opening) / same with `visibility 0s linear 0.3s` (closed). The 300ms + easing are **copied from Ollie Pro's** `.wp-block-group[data-sticky-on-scroll-up="true"] { transition: transform 0.3s cubic-bezier(0.4, 0, 0.2, 1) }` – CSS can't read another element's timing, so it's a duplicated constant; header and panel must always share it. Ollie fades the panel over ~100ms, the header over 300ms, so over a transparent header they arrive out of sync. Scoped to `:not(.is-scrolled)`: only a *fading* header (transparent, turning solid) has a mismatch; a scrolled solid header keeps Ollie's snappier default. Two states because `visibility` flips at once on the way in (usable immediately) and waits the fade on the way out (stays visible while fading). No `!important` – the scoped selector already outranks Ollie's.
  - **Overlap removal.** `top: var(--kntnt-mega-menu-top, 100%) !important` on the panel. Ollie sets the panel's `top` from its *Top spacing* as an **inline style via JS**, landing it ~25px inside the header's padding; the two same-colour surfaces overlap and composite to a bright band mid-fade (`1-(1-a)^2 > a`). Syncing the fades is what *reveals* the band; removing the overlap is what removes it. `js/mega-menu-offset.js` measures the header group's live bottom edge minus the panel's offset-parent top into `--kntnt-mega-menu-top`; the rule hangs the panel from there. `!important` is unavoidable (beats Ollie's inline `top`); the custom-property indirection is deliberate – Ollie rewrites its inline `top` on resize but never touches the property, so the stylesheet keeps winning without a race. `100%` fallback hangs the panel from the nav item's own bottom before JS runs.
- `js/mega-menu-offset.js` writes `--kntnt-mega-menu-top` on menu open (synchronously, before paint – no fallback flash) and on resize; skips the mobile overlay (`position: fixed`). Setting the top *before* the panel paints is the **handshake** with `kntnt-modal-mega-menu-ollie`: that plugin caps a tall menu's height ~100ms after open by measuring the panel's live top, so it reads the corrected edge with no shared event needed (the "guaranteed ordering" its `agents.d/modal.md` calls for). Loaded for every visitor (the band is visible to all), self-guarding on pages with no transparent header or no mega menu. Like the admin-bar offset, the whole thing is a **kludge replaceable by CSS anchor positioning** once Safari ships it – see `removable-workarounds.md`.
